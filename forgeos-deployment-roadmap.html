The 4 Agents

Design constraint: Employees already have access to chat-based AI tools. These 4 agents are autonomous, always-on, event-driven, and scheduled — they work in the background without human prompting, surfacing results and requesting approval only when needed.

Client Data Analyzer Scheduled Event-Driven

Runs scheduled analyses on client data repositories. Generates reports, surfaces insights, flags anomalies. Can also be triggered on-demand by employees.

Output: Analysis reports pushed to employees. Read-only — never modifies source data.

Invoice Processor Event-Driven Scheduled

Processes incoming invoices from clients — extracts data (amounts, vendor, line items, GL accounts) and uploads structured records into SAP.

Output: SAP document entries. Financial write operations — HITL-gated.

Email/Enquiry Responder Event-Driven

Monitors inbound client emails. Classifies intent and urgency. Drafts contextual responses using client history and knowledge base. Sends approved responses.

Output: Client-facing emails. Highest risk — brand-representing, irreversible.

Activity Manager Always-On Event-Driven

Continuously monitors client activities. Updates Jira tickets, syncs team calendars, tracks deadlines, coordinates handoffs. The operational backbone.

Output: Jira updates + calendar events. Multi-system writes, highest coordination complexity.

The Graduated Autonomy Model

Every agent follows the same 4-stage autonomy ramp. No agent goes from zero to autonomous. The progression is data-driven (accuracy metrics), not time-driven (arbitrary deadlines).

Phase 0: Infrastructure (Week 0)

Before ANY agent deploys, this infrastructure must be operational. Begin provisioning in parallel with Phase 1 Week 1.

Authentication & MCP Connections

  • Google Workspace OAuth — Service account for Gmail, Calendar, Drive, Sheets (all 4 agents need this)
  • SAP OData credentials — SAP_IAS_URL, SAP_IAS_CLIENT_ID, SAP_IAS_CLIENT_SECRET (begin now — SAP auth is slow)
  • Atlassian API token — Jira + Confluence access for Activity Manager
  • CRM API keys — Read access for Data Analyzer, read/write for others

Platform Deployment

  • HITL Dashboard — Deploy ForgeOS dashboard at client site. Ensure /api/approvals/{id} is accessible
  • Slack Integration — Configure #approvals, #incidents channels for hook notifications
  • PostgreSQL — For audit logs, sessions, knowledge base, ontology
  • Monitoring — AuditLogger + MetricsStore + SlackNotifier operational

Knowledge Base Seeding

  • Vendor master data (names, SAP codes, GL accounts)
  • Client profiles (history, SLAs, contract terms, contacts)
  • Email response templates (50+ approved responses by category)
  • Escalation policies (who approves what, when to escalate)
  • Team member profiles (roles, calendars, Jira usernames, preferences)

Ontology Schema

  • Customer — name, industry, tier, SLA, contacts
  • Invoice — vendor, amount, GL account, status, SAP doc number
  • Enquiry — client, intent, urgency, status, response_time
  • Activity — title, type, owner, status, jira_key, calendar_event_id
  • TeamMember — name, role, department, calendar, jira_username

Phase 1: Foundation + Quick Wins

Weeks 1-4 — Client Data Analyzer (Scheduled + Event-Driven)

Why start here: Lowest complexity (13/40). Read-only operations — zero write risk. Internal-only — no client-facing exposure. Failure cost is near zero (bad report gets discarded). Proves the entire infrastructure works before touching financial or client-facing systems.
Week 1 — Shadow Mode

Scaffold + First Run on Real Data

  • Deploy agent using PlatformExecutor.deploy() pipeline
  • Adapt the client-reporter practical agent prompts to target client's specific data repository
  • Configure scheduled execution: daily at 8 AM (SchedulerEngine with cron "0 8 * * *")
  • Run against 3-5 client accounts in shadow mode — outputs generated but NOT distributed
  • Humans compare agent reports vs their own analysis side-by-side
Week 2 — Shadow Validation

Tune & Expand

  • Measure accuracy: do the numbers match? Are insights relevant? Are there hallucinated statistics?
  • Tune system prompt based on discrepancies (enforce: "Every number must come from a tool call")
  • Seed KnowledgeBase with corrections from shadow mode comparisons
  • Track via MetricsStore: analyzer.accuracy, analyzer.coverage, analyzer.hallucination_count
  • Gate: >85% accuracy alignment with human-generated reports before proceeding
Week 3 — Supervised Mode

Employees See the Output

  • Switch to supervised mode — reports generated and shown to employees, marked "AI-generated"
  • Enable on-demand trigger: employees fire analysis.requested events via dashboard button
  • Expand to 10+ client accounts
  • Track usage: how often employees trigger analyses, which report types they value most
Week 4 — Production

Full Rollout + Feedback Loop

  • Production deployment across all client accounts
  • Recurring schedules: weekly summaries (Monday 8 AM), monthly deep dives (1st of month)
  • Establish feedback loop: employees rate quality, corrections feed into KnowledgeBase
  • Add event subscription: analysis.completed notifies relevant teams via Slack

Phase 2: Event-Driven Intelligence

Weeks 5-10 — Invoice Processor (Event-Driven + Scheduled)

Why this order: Invoice processing scores 26/40 — the first agent that writes data. But invoices are structured data with well-defined schemas, and SAP entries can be corrected via credit memos. The failure mode is expensive but bounded (wrong posting, not lost client). This builds the HITL muscle in the finance team before we tackle client-facing operations.
Week 5 — SAP Integration

Connect SAP + Scaffold Agent

  • Deploy and validate SAP OData MCP connection (btp-sap-odata-to-mcp-server)
  • Test read operations first: query existing invoices, vendor master, GL accounts
  • Scaffold the Invoice Processor extending invoice-categorizer practical agent
  • Build vendor mapping table in KnowledgeBase: vendor name variations → canonical vendor ID → SAP code
Weeks 6-7 — Shadow Mode

Extraction Pipeline on Real Invoices

  • Process real invoices but do NOT write to SAP
  • Build extraction pipeline: Email/attachment intake → PDF parsing → field extraction → validation → structured output
  • Run against 50+ historical invoices to benchmark per-field accuracy
  • Track: invoice number, date, amount, vendor, line items, tax code, GL account accuracy individually
  • Implement dual-extraction: extract twice with different prompts, flag discrepancies for human review
  • Implement idempotency: hash(invoice_number + vendor + amount + date) prevents duplicate processing
Weeks 8-9 — HITL-Gated SAP Writes

100% Human Approval on Every Write

  • Enable SAP writes with 100% HITL approval
  • Every extraction creates an ApprovalRequest via HITLGateway — category: "financial", SLA: 24 hours
  • Slack notification sends approvers a dashboard link with extracted data side-by-side with original invoice image
  • Track approval/rejection rates and reasons — rejections feed into prompt tuning
  • Target: >90% approval rate by end of Week 9
Week 10 — Confidence-Based Autonomy

Threshold-Based Auto-Approval

  • Auto-approve: confidence ≥95% AND amount <$5,000 AND known vendor
  • Batch review (4h SLA): confidence ≥90% OR amount $5,000-$10,000
  • Mandatory individual HITL: confidence <90% OR amount >$10,000 OR new vendor
  • Set up EventBus subscription for invoice.received events — fully automated trigger
  • All duplicate-flagged invoices always require HITL

Phase 3: Full Autonomy

Weeks 11-16+ — Email Responder + Activity Manager (in parallel)

Why parallel: These two agents have independent integration surfaces (email vs Jira/Calendar) and can be developed by separate team members. Both score 32-36/40 but for different reasons — the Responder's risk is irreversibility + client-facing; the Activity Manager's risk is multi-system coordination + business context depth.

Track A: Email/Enquiry Responder

Weeks 11-12 — Classification Pipeline (Shadow)

Classify + Draft, But Humans Send

  • Deploy email-triage practical agent as intake stage (already production-tested for classification)
  • Build response drafting agent adapted from ticket-router (FAQ response via knowledge base)
  • Shadow mode: agent classifies emails and drafts responses, but a human sends them
  • Seed KnowledgeBase with 50+ approved response templates by enquiry type
  • Track classification accuracy and draft quality scores
Weeks 13-14 — HITL-Gated Responses

Every Response Approved Before Sending

  • Full pipeline: classify → draft response → HITLGateway (category: "content", 4h SLA) → send
  • ComplianceChecker validates every outbound email before it reaches the HITL queue — blocked emails never reach approvers
  • Track: approval rate, edit distance between draft and final version, client satisfaction
  • Every approved response (without edits) gets added to KnowledgeBase as a "golden template"
Weeks 15-16 — Graduated Autonomy

Auto-Send High-Confidence, HITL the Rest

Auto-send when ALL criteria met:

  • Enquiry type matches a known template (>95% similarity to approved golden template)
  • Client is NOT flagged as high-value or high-risk
  • Response does NOT mention money, contracts, legal terms, or commitments
  • Classification confidence >90%

All other responses continue through HITL. Random 10% audit sampling of auto-sent responses.

Cooling-off rule: 3 rejections in a row → agent pauses and escalates to engineering.

Track B: Activity Manager

Weeks 11-12 — Read-Only Mode

Monitor Everything, Write Nothing

  • Deploy and validate mcp-atlassian MCP connection for Jira
  • Agent monitors Jira and Google Calendar in read-only mode
  • Generates daily activity summaries but does NOT create/update tickets or events
  • Populate TeamMember ontology with profiles, roles, preferences
  • Build project knowledge base: escalation rules, SLA definitions, priority mapping
  • Validate always-on loop: ExecutionType.ALWAYS_ON with loop_interval_seconds: 60
Weeks 13-14 — HITL-Gated Writes

Jira + Calendar Writes With Approval

  • Enable Jira writes (create ticket, update status, add comment) — all HITL-approved
  • Enable Google Calendar writes (create event, modify event) — all HITL-approved
  • Build deduplication: detect when Jira webhooks create feedback loops (agent updates ticket → webhook fires → agent tries again)
  • Rule: always call query_freebusy before create_event — if conflict, queue for HITL instead of auto-resolving
Weeks 15-16 — Supervised Autonomy

Auto-Approve Low-Risk, HITL High-Risk

Auto-approve:

  • Status updates on tickets assigned to the agent's scope
  • Adding comments to Jira tickets
  • Internal-only 1:1 meetings with confirmed free time slots

Always HITL:

  • Multi-person meetings, client-facing calendar events
  • Priority escalation to Critical/Blocker
  • Ticket reassignment, any action affecting >3 team members

Enable cross-agent events: Invoice Processor's invoice.processed → Activity Manager updates related Jira ticket. Email Responder's enquiry.resolved → Activity Manager closes related activity.

Cross-Agent Events (Phase 3+)

Once all 4 agents are operational, they communicate via the EventBus:

Invoice Processor                     Activity Manager
  publishes: invoice.processed  ───→   subscribes: updates related Jira ticket
  publishes: invoice.failed     ───→   subscribes: creates incident ticket

Email Responder                       Activity Manager
  publishes: enquiry.resolved   ───→   subscribes: closes related activity
  publishes: enquiry.escalated  ───→   subscribes: creates high-priority Jira ticket

Data Analyzer                         Email Responder
  publishes: anomaly.detected   ───→   subscribes: drafts proactive client notification

Activity Manager                      Data Analyzer
  publishes: deadline.missed    ───→   subscribes: includes in next client report
The compounding effect: Each agent makes the others better. The Data Analyzer surfaces insights that the Email Responder uses for proactive client communication. The Invoice Processor's completions update the Activity Manager's project tracking. The Activity Manager's deadline alerts feed into the Data Analyzer's client health scores. By Week 16, you have an integrated autonomous operations layer, not 4 isolated bots.
