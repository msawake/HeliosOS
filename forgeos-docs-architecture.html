Three-Layer Design

ForgeOS is built in three layers with clear boundaries enforced by Python package structure. Each layer has a single responsibility and communicates through well-defined interfaces.

1

Layer 1: Stack Adapters stacks/

Provide different agent runtimes. Each adapter wraps a different AI framework behind a common AgentStackAdapter interface. Different frameworks have different strengths — ForgeOS supports all four and lets each agent pick the best fit.

ForgeOS Native
CrewAI
Google ADK
OpenClaw
# stacks/base.py -- The universal adapter interface
class AgentStackAdapter(ABC):
    async def create_agent(self, agent_def: AgentDefinition) -> str
    async def invoke(self, agent_id: str, prompt: str, ...) -> AgentResult
    async def start_loop(self, agent_id: str) -> None
    async def stop(self, agent_id: str) -> None
    def get_status(self, agent_id: str) -> AgentStatus
    def scaffold_files(self, agent_def: AgentDefinition) -> dict[str, str]
Fallback pattern: When a framework SDK is not installed (e.g., crewai package missing), the adapter automatically falls back to the ForgeOS native agentic loop. Every agent works out of the box.

Boot Sequence

When ForgeOS starts via python -m src.bootstrap, these 15 steps execute in order. The entire boot is wrapped in try/except with stage-tracked cleanup.

1
Load .env Environment

Read API keys, DATABASE_URL, REDIS_URL, and all FORGEOS_* configuration variables from the .env file.

2
Initialize LLM Router Environment

Detect available providers (Anthropic, OpenAI) based on configured API keys. Build provider clients.

3
Connect Database Infrastructure

Establish PostgreSQL connection pool with Row-Level Security. Falls back to in-memory if DATABASE_URL is not set.

4
Run Migrations Infrastructure

Apply SQL migration files from infrastructure/database/. Creates tables, indexes, and RLS policies.

5
Initialize CompanySystem Infrastructure

Wire up event bus, HITL gateway, knowledge base, and metrics subsystem. Shared infrastructure for all company packages.

6
Connect MCP Servers Infrastructure

Spawn stdio processes for configured MCP servers. Discover tools via list_tools() with a 30-second timeout per server.

7
Create ToolExecutor Infrastructure

Register MCP tools, custom company tools, and platform tools into the unified tool executor for routing.

8
Wire UsageEnforcer Infrastructure

Initialize token and cost tracking per tenant. Connect to billing subsystem for limit enforcement.

9
Register Stack Adapters Platform

Initialize and register all four stack adapters: ForgeOS, CrewAI, ADK, OpenClaw. Each adapter auto-detects its SDK.

10
Build PlatformExecutor Platform

Connect registry, scheduler, event bus, and all adapters into the central executor. This is the orchestration core.

11
Recover Agents Platform

Load persisted agents from database. Re-wire execution lifecycles for non-terminal agents. Quarantined agents are skipped.

12
Create Tenant Data

Ensure the default tenant exists in the database to satisfy foreign key constraints on all tenant-scoped tables.

13
Seed Data Data

Populate knowledge base entries, sample HITL approval records, and initial configuration for company packages.

14
Start Scheduler Runtime

Begin cron job execution. APScheduler starts its event loop; fallback mode spawns asyncio tasks for each registered job.

15
Start API Server Runtime

Launch FastAPI on uvicorn in a background thread. The platform is now accepting HTTP requests and streaming connections.

Execution Type Wiring

When an agent is deployed, the executor's _wire_execution method connects the agent to the correct lifecycle mechanism based on its execution type.

Execution TypeWiring ActionTrigger
always_onCalls adapter.start_loop(agent_id), sets status to RUNNINGContinuous — runs immediately
scheduledAdds job to SchedulerEngine with cron expressionCron schedule fires callback
event_drivenSubscribes to each trigger via EventBusMatching event fires callback

# PlatformExecutor.deploy() sequence

1. Validate    Name must match ^[a-zA-Z][a-zA-Z0-9_-]{1,63}$
               Path traversal characters are rejected

2. Register    registry.register(agent_def)
               Raises ValueError on duplicate agent_id

3. Scaffold    adapter.scaffold_files(agent_def)

4. Create      adapter.create_agent(agent_def)
               Stack-specific initialization

5. Wire        _wire_execution(agent_def)
               Connect to lifecycle mechanism (see table above)

# On failure at any step: registration is rolled back, scaffolded files deleted

Autonomous Loop Recovery

Autonomous agents use exponential crash backoff. After consecutive failures exceed the threshold, the agent is quarantined and requires manual intervention.

Config KeyDefaultDescription
max_iterations50Maximum loop iterations before stopping
loop_interval_seconds30Sleep between iterations (seconds)
restart_on_failureFalseIf True, FAILED iterations do not terminate the loop
max_crashes_before_give_up3Consecutive crashes before QUARANTINED status
Crash backoff formula: min(60, 2^crash_count) seconds. After max_crashes_before_give_up consecutive crashes the agent is set to QUARANTINED — it will not be auto-recovered on restart.

Retry Configuration

Environment VariableDefaultDescription
FORGEOS_LLM_MAX_RETRIES3Maximum retry attempts per call
FORGEOS_LLM_BACKOFF_BASE1.0Base delay in seconds
FORGEOS_LLM_BACKOFF_MAX30.0Maximum delay cap in seconds

Transient errors (HTTP 429, 5xx, timeouts) trigger retry with exponential backoff plus jitter. Non-retryable errors (4xx except 408/429) fail immediately. When the primary provider fails and fallback_provider is set in metadata, a single failover attempt is made.
