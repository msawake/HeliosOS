The Surprising Truth

170 agents use less RAM than a single Chrome tab and less CPU than Slack.
The bottleneck is never your hardware. It's your Anthropic API plan and your willingness to pay for tokens.

ForgeOS agents are asyncio coroutines — they sleep 95% of the time. When one agent waits for an LLM response, the event loop runs other agents. The machine barely notices. Here are the 5 things that actually limit you, ranked by which hits first:

The 5 Bottlenecks (ranked)

Hits first. Always. Regardless of hardware.

#3
Thread Pool
Python defaults to min(32, cpu_count + 4) worker threads.

Each agent making an LLM call occupies one worker thread. Python's default pool:

*Raise with: loop.set_default_executor(ThreadPoolExecutor(max_workers=64))

In practice this rarely matters. With 40 agents on 60s cycles, each active ~15s → only ~10 threads used at peak. Agents naturally desync.

#4
Memory
170 agents = ~300 MB. Your Mac has 16+ GB.

Each agent session stores: system prompt (~3 KB) + conversation history (~100 KB) + tool results (~30 KB) + Python overhead (~2 KB) = ~100-300 KB per session.

#5
CPU
Almost never the bottleneck. 40 agents = <5% CPU.

The event loop just schedules coroutines (near-zero CPU). LLM calls are network I/O (zero local CPU). Tool execution is dict lookups and JSON serialization. Hook chain is 7 O(1) checks.

The only CPU-intensive scenario would be local document processing — which ForgeOS doesn't do. All intelligence lives in the LLM API.

7 Key Hypotheses

1
API rate limits are the #1 bottleneck — not CPU, RAM, or threads. A $100/month Anthropic account limits you to ~8-15 concurrent agents regardless of whether you're on a $999 MacBook or a $50,000 server.
2
Cost is the #2 bottleneck. An Opus agent running always-on at 60-second cycles costs ~$1,080/day. Production deployments must use longer cycles (5-30 min) or event-triggered patterns to stay sane.
3
A Mac laptop can run a full AI company — 37 registered agents, 8-15 active simultaneously — with a Build-tier API key and ~$130/day budget.
4
Upgrading hardware doesn't help. Upgrading your API tier does. The jump from laptop to server is about rate limits (Build → Scale), not about CPU or RAM.
5
Memory and CPU are irrelevant. 170 agents use ~300 MB of RAM (less than a Chrome tab) and <5% CPU. The machine is practically idle.
6
Thread pool can bottleneck on laptops if >12 agents make LLM calls simultaneously, but natural desync prevents this. Agents don't all wake up at the same time because LLM call durations vary.
7
Multi-tenant scales linearly. Each tenant is an independent company. 10 tenants × 40 agents = 400 agents, limited only by the Enterprise API tier.

How to Run More Agents (Same Hardware, Same Budget)

Increase Cycle Intervals

5-minute cycles instead of 60-second cycles.

5x more agents for the same API rate
Use Event-Triggered over Always-On

Zero cost when idle. Agent sleeps until a signal fires.

90%+ cost reduction for sporadic workloads
Use Haiku for Simple Tasks

Simple Reflex agents don't need Opus reasoning power.

15x cheaper per cycle vs Opus
Batch Agent Work

One Opus call doing 10 tasks beats 10 Haiku calls doing 1 each.

Better token efficiency per dollar
Increase Thread Pool

set_default_executor(ThreadPoolExecutor(64))

2x concurrent LLM calls on laptops
Mix Model Tiers Smartly

Haiku for monitoring, Sonnet for workflows, Opus only for decisions.

3-5x cost reduction with same output quality
