Agents

Execution types, ownership, lifecycle, tools, and deployment v3.0

What is an Agent?

An agent is a configured AI workload. It does not exist as standalone code — it runs inside the ForgeOS framework. You declare what an agent should do (name, stack, execution type, tools, system prompt), and the platform handles everything else: provisioning files, registering in the registry, wiring its execution lifecycle, and routing every LLM call through the configured provider.

Key distinction: The framework is the platform — it boots once and runs forever. Agents are the programs — they are deployed, invoked, and managed by the framework. The framework does not perform business tasks; agents do.

The 5 Execution Types

Execution type determines an agent's lifecycle — how and when it runs. Choose the type that matches your workload pattern.

⏻
always_on
Runs continuously in a persistent loop from deployment until explicitly stopped.
Uptime sentinel
⏰
scheduled
Triggered by a cron expression. Stays idle between runs.
Nightly lead scoring
⚡
event_driven
Subscribes to named events on the platform event bus. Invoked when events fire.
Deal alert notifier
⇄
reflex
On-demand only. Responds to direct API calls or dashboard chat. The simplest type.
Chat assistant
🎯
autonomous
Goal-directed loop. Iterates until it completes its objective or exhausts its budget.
Market research

The 3 Ownership Types

Ownership determines where the agent's files are scaffolded on disk and who can access it.

👤

personal

Scoped to a single user. Only visible to the owner. Requires owner_id.

agents/personal/{owner_id}/{name}/
👥

shared

Company-wide. Visible to all users in the tenant. The default for most production agents.

agents/shared/{name}/
🏢

client

Per-client isolation. Each client gets its own directory with isolated tool configs. No cross-client data leakage.

agents/clients/{client_id}/{name}/

Quarantine safety: When an autonomous agent throws unhandled exceptions on consecutive iterations and hits the crash threshold (default: 3), the platform quarantines it. A quarantined agent will not be re-wired during recover(). To fix: investigate logs, undeploy, and redeploy with corrected configuration.

Tools & MCP

Agents gain capabilities through tools. When an agent runs, the LLM can request tool calls, and ForgeOS executes them and returns results. All calls route through a single ToolExecutor which enforces per-agent whitelists and provides a unified audit trail.

Tool Wildcards

Use * suffix to match tool groups. This is useful for granting access to all tools from a particular MCP server or domain.

PatternMatches

[] (empty)All available tools (not recommended for production)

Tool Execution Flow

1. LLM returns a tool_use block  {"name": "mcp__filesystem__read_file", "input": {...}}
2. Agentic loop calls _execute_tool(name, input, context)
3. Retry wrapper: 2 attempts, per-tool timeout (60s default)
4. ToolExecutor.execute() checks whitelist, then routes:
     company__*  →  in-process Python handler (sync)
     mcp__*      →  MCP server session.call_tool() (async, 120s timeout)
     unknown     →  {"success": false, "error": "Unknown tool"}
5. Result serialized to JSON → appended to messages as tool_result
6. LLM sees the result and decides next action

Deploy & Invoke Examples

Deploy a Reflex Chat Agent

curl -X POST http://localhost:5001/api/platform/agents \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $FORGEOS_TOKEN" \
  -d '{
    "name": "support-chatbot",
    "stack": "forgeos",
    "execution_type": "reflex",
    "ownership": "shared",
    "description": "Answers customer support questions using company knowledge base.",
    "system_prompt": "You are a helpful support agent for Acme Corp. Answer questions about billing, shipping, and returns. Be concise and accurate.",
    "chat_model": "claude-4-sonnet",
    "provider": "anthropic",
    "tools": ["knowledge__search", "knowledge__lookup"]
  }'

Response (HTTP 201):

{
  "agent_id": "a1b2c3d4e5f6",
  "name": "support-chatbot",
  "stack": "forgeos"
}

SSE Event Types

EventDescription
sessionFirst event. Contains session_id for the conversation.
text_deltaIncremental text chunks as the LLM generates its response.
tool_callEmitted when the agent invokes a tool. Contains tool name and input.
tool_resultThe result returned by the tool.
hitl_requestHuman-in-the-loop approval request (if the agent needs confirmation).
doneFinal event. Contains total tokens_used and complete text.
errorEmitted if something goes wrong. Contains an error string.

Stop an Agent

curl -X POST http://localhost:5001/api/platform/agents/{agent_id}/stop \
  -H "Authorization: Bearer $FORGEOS_TOKEN"

Cancels active loops, removes scheduled jobs, removes event subscriptions, and sets status to STOPPED.

Undeploy (Delete) an Agent

curl -X DELETE http://localhost:5001/api/platform/agents/{agent_id} \
  -H "Authorization: Bearer $FORGEOS_TOKEN"

Stops the agent, deletes scaffolded files from agents/, and unregisters from the platform. This is irreversible.

Best Practices

Agent Design

  • Keep system prompts focused and specific about role, tools, and output format
  • Start with reflex — test interactively before switching to other types
  • Set clear, verifiable goals for autonomous agents
  • Name agents descriptively in kebab-case: daily-pipeline-report
  • Prefer shared over personal for team-serving agents

Operational Safety

  • Use explicit tool lists — avoid broad wildcards in production
  • Test before scheduling — invoke manually first to verify output
  • Tune autonomous metadata: max_iterations, loop_interval_seconds
  • Monitor quarantined agents — they require manual intervention
  • Use metadata for tenant context: tenant_id, plan, monthly_limit_usd
