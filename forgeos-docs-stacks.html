The Common Interface

Every stack adapter implements AgentStackAdapter from stacks/base.py. The platform executor, scheduler, and event bus call these methods uniformly — they never reach into adapter internals. When a stack's SDK is unavailable, each adapter gracefully falls back to the ForgeOS native agentic loop. 

class AgentStackAdapter(ABC):
    """Interface that every agent stack must implement."""

    stack_name: str  # "forgeos", "crewai", "adk", "openclaw"

    async def create_agent(self, agent_def: AgentDefinition) -> str:
        """Provision an agent from the definition. Returns agent_id."""

    async def invoke(self, agent_id: str, prompt: str,
                     context: dict | None = None,
                     history: list[dict] | None = None) -> AgentResult:
        """Run a single invocation of the agent."""

    async def start_loop(self, agent_id: str) -> None:
        """Start a persistent loop (always-on / autonomous agents)."""

    async def stop(self, agent_id: str) -> None:
        """Stop a running agent."""

    def get_status(self, agent_id: str) -> AgentStatus:
        """Return the current status of an agent."""

    def scaffold_files(self, agent_def: AgentDefinition) -> dict[str, str]:
        """Return {relative_path: file_contents} for scaffold templates."""

    async def recover(self) -> int:
        """Optional: stack-specific recovery after boot. Returns count."""
Uniform dispatch: The platform executor calls adapter.invoke(agent_id, prompt) regardless of whether the agent runs on ForgeOS, CrewAI, ADK, or OpenClaw. The adapter handles SDK initialization, tool bridging, and fallback internally.

ForgeOS Native

Native   ForgeOS Adapter

The default and simplest adapter. Uses the platform's own agentic loop directly with zero external dependencies.

invoke() → build_tool_definitions() → run_agentic_loop() → LLM ↔ tool_executor → AgentResult LLM ↔ tools loop until done

How It Works

  • Create: Stores the AgentDefinition in an internal dict. No SDK initialization needed — the agent is immediately ready.
  • Invoke: Builds tool definitions from tool_executor, constructs a system prompt from the agent definition, then runs the LLM → tool_use → execute → tool_result → LLM loop until complete.
  • Tools: Passed directly to the LLM as Anthropic-format tool definitions. Tool calls route through tool_executor.execute() with zero wrapping overhead.
When to use: Default choice for all agents. No SDK dependency. Full access to all platform features. Best for reflex agents, scheduled agents, and any agent that primarily uses MCP tools.

Pros

  • Zero external dependencies
  • Full access to all platform features
  • Direct tool execution (no bridging overhead)
  • Native multi-turn history support
  • This IS the fallback — always available

Cons

  • Single-agent only (no built-in crew/delegation)
  • No native workflow composition primitives
  • Tied to the ForgeOS agentic loop pattern

CrewAI

Crews   CrewAI Adapter

Wraps the CrewAI framework for role-based agent collaboration. Agents are defined with roles, goals, and backstories, then orchestrated by a Crew.

invoke() → Crew.kickoff() → thread pool → CrewAgent → BaseTool._run() → new event loop → tool_executor

Tool Bridging

CrewAI tools must be BaseTool subclasses with a _run(**kwargs) -> str method. The adapter creates a wrapper class for each ForgeOS tool. A new event loop is created per tool call because CrewAI runs in a worker thread (via run_in_executor), which has no active asyncio loop.

class ForgeOSTool(CrewBaseTool):
    name: str = "mcp__google-workspace__send_gmail_message"
    description: str = "Send email via Gmail API"

    def _run(self, **kwargs) -> str:
        loop = asyncio.new_event_loop()
        try:
            result = loop.run_until_complete(
                tool_executor.execute(self.name, kwargs, agent_context)
            )
        finally:
            loop.close()
        return str(result)
When to use: When agents need explicit role definitions and you want CrewAI's built-in delegation patterns. Good for multi-step workflows where the "crew" metaphor fits (e.g., a researcher + writer + editor pipeline).

Pros

  • Role-based collaboration out of the box
  • Built-in delegation and task sequencing
  • Rich ecosystem of CrewAI community tools
  • LiteLLM auto-detects model providers

Cons

  • Requires pip install crewai
  • Blocking kickoff() runs in thread pool
  • New event loop per tool call (overhead)
  • History injected as text (not native turns)

Google ADK

ADK   Google ADK Adapter

Wraps Google's Agent Development Kit for enterprise workflows. Supports native Gemini models, with Anthropic and OpenAI via the model factory.

invoke() → Runner.run_async() → event stream → extract text / tools / tokens → AgentResult

Model Factory

The _build_adk_model() function routes model strings to the correct ADK model class:

Model PrefixADK ModelFallback
claude-*AnthropicLlm(model=...)LiteLlm("anthropic/...")
gpt-* / o3-* / o1-*LiteLlm("openai/...")—
gemini-*Bare string (ADK native)—

Tool Bridging

ADK tools are FunctionTool instances wrapping async functions. ADK inspects the function signature to build a schema, so the wrapper accepts **kwargs.

async def _wrapper(**kwargs):
    """Run the ForgeOS tool and return the raw result dict."""
    result = await tool_executor.execute(tool_name, kwargs, agent_context)
    return result if isinstance(result, dict) else {"result": str(result)}

tool = FunctionTool(_wrapper)
When to use: When integrating with Google Cloud services, using Gemini models natively, or when you need ADK's built-in SequentialAgent / ParallelAgent composition patterns for multi-agent orchestration.

Pros

  • Native Gemini support (bare string, zero wrapping)
  • Async event stream architecture
  • Built-in SequentialAgent / ParallelAgent
  • InMemorySessionService for state management
  • Multi-model via AnthropicLlm + LiteLlm

Cons

  • Requires pip install google-adk
  • Heavier SDK footprint
  • Agent names must be valid identifiers (no dashes)
  • Content objects require google.genai.types

OpenClaw

Gateway   OpenClaw Adapter

Wraps the OpenClaw framework, which uses a file-first approach with markdown-based agent configuration. Agents are defined by SOUL.md personality files and communicate via an HTTP gateway subprocess.

invoke() → _ensure_gateway() → HTTP POST → /v1/chat/completions → parse response → AgentResult

Gateway Lifecycle

The OpenClawGateway class manages a Node.js subprocess serving an OpenAI-compatible REST API:

start() → node openclaw.mjs gateway → _wait_for_ready() → poll /health (0.5s, 30s timeout) → chat() → stop() → SIGTERM → SIGKILL (5s)

The gateway auto-restarts after crashes (no one-shot gate). Start is protected by asyncio.Lock to prevent double-start.

Workspace Files

Each agent gets a workspace directory at ~/.openclaw-forgeos/workspaces/{agent-name}/:

FilePurpose
SOUL.mdSystem prompt and personality definition
AGENTS.mdAgent metadata, department, tools list
HEARTBEAT.mdSchedule, cron triggers, event triggers
memory.mdPersistent memory (updated by agent at runtime)
When to use: When you want markdown-driven agent definitions (SOUL.md pattern), or when integrating with the OpenClaw ecosystem. Good for agents that emphasize memory and personality over tool use.

Pros

  • Markdown-first — edit agents in any text editor
  • Persistent memory built into the workspace
  • OpenAI-compatible API (standard protocol)
  • Gateway auto-restarts on crash
  • Workspace recovery on boot via recover()

Cons

  • Requires Node.js + openclaw2 directory
  • HTTP subprocess adds latency vs in-process
  • Tools configured in AGENTS.md (not code-level bridging)
  • 30s startup timeout for gateway health check

Adding a New Stack

To add a fifth stack adapter (e.g., LangGraph), follow these 6 steps. The platform layer, dashboard, and API will automatically discover the new stack.

  1. Create the adapter module. Add stacks/langgraph/__init__.py and stacks/langgraph/adapter.py with a class that extends AgentStackAdapter.
  2. Implement all 6 required methods. create_agent(), invoke(), start_loop(), stop(), get_status(), and scaffold_files(). Optionally override recover() for boot-time state rebuilding.
  3. Set the stack name. Define stack_name = "langgraph" as a class attribute. This string is used for routing, registry lookups, and API responses.
  4. Register the stack name. Add "langgraph" to the STACK_NAMES tuple in stacks/base.py. This enables validation in AgentDefinition.__post_init__().
  5. Register in bootstrap. Add the adapter to src/bootstrap.py in the _register_adapters() function so the platform executor discovers it at boot.
  6. Implement scaffold_files(). Return a dict[str, str] of file templates matching the new framework's project layout. The dashboard and API use this to generate agent code on deployment.
Graceful degradation: Follow the pattern used by CrewAI and ADK — detect the SDK at import time, store a boolean flag (e.g., LANGGRAPH_AVAILABLE), and fall back to run_agentic_loop() when the SDK is missing. This ensures agents always run, even without the external dependency.
