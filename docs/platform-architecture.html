1. Executive overview

ForgeOS 3.x is a multi-stack agent platform. It presents one uniform way to define, deploy, schedule, invoke, and tear down agents, while supporting four different frameworks under the hood: native ForgeOS, CrewAI, Google ADK-style scaffolds, and OpenClaw-style file-first agents. The same core abstractions (AgentDefinition, AgentStackAdapter, PlatformExecutor) apply to every stack.

Agents can run in five execution modes (always-on, scheduled, event-driven, reflex, autonomous) and can be personal (per-user) or shared (team / enterprise-wide; in code this is OwnershipType.SHARED).

The runtime combines a Python backend (bootstrap, Flask API, platform services, legacy company agents) with a Next.js 15 developer dashboard that proxies API calls to Flask in development.

3. Bootstrap and startup phases

PlatformBootstrap in src/bootstrap.py orchestrates startup. The high-level sequence is:

PhasePurpose
1 — Platform Build LLMRouter from environment API keys (ANTHROPIC_API_KEY, OPENAI_API_KEY). Without keys, chat calls are simulated for stacks and wizard heuristics can still run.
2 — Legacy company CompanySystem, database client (PostgreSQL or in-memory), MCP manager, ToolExecutor, ClaudeClient, company module registry, AgentInvoker for classic departmental agents.
3 — Adapters Instantiate ForgeOSAdapter, CrewAIAdapter, ADKAdapter, OpenClawAdapter, each wired with llm_router (and ForgeOS with tool executor).
4 — Executor Create PlatformExecutor sharing registry, scheduler, and platform EventBus; register all adapters.
5 — Knowledge Seed company knowledge base when CompanySystem is present.
HITL seed (dev) Optional demo approvals when FORGEOS_SEED_HITL is not disabled.
6 — Workflow engine WorkflowEngine tied to legacy invoker (Flask /api/workflows).
7 — Scheduler start scheduler.start_all() activates asyncio jobs already registered for platform agents.

Starting the HTTP API: python -m src.bootstrap --dashboard (ideally with project venv and PYTHONPATH=.). The process stays alive while Flask serves on port 5000 by default.

4. Platform layer

4.1 AgentRegistry

File: src/platform/registry.py. In-memory store of AgentDefinition instances and AgentStatus. Supports listing, filtering by stack, execution type, ownership, department, owner id, and summary counts for the dashboard overview.

4.2 PlatformExecutor

File: src/platform/executor.py. Central operations:

  • deploy — resolve directory, scaffold via adapter, register, create_agent, then _wire_execution.
  • invoke — route prompt to the correct adapter.
  • stop_agent — stop adapter loops, cancel autonomous tasks, remove scheduler job, unsubscribe platform events.
  • undeploy — stop, delete agent directory, unregister.

4.3 SchedulerEngine

File: src/platform/scheduler.py. MVP interval parser: strings like every 15m, every 1h, or minute-style */N * * * * mapped to seconds. Each scheduled agent gets an asyncio loop that sleeps then invokes a callback (typically invoke with a schedule string in the prompt).

4.4 EventBus (platform)

File: src/platform/event_bus.py. Separate from the company EventBus in MCP tools. Agents with EVENT_DRIVEN execution subscribe to named events; POST /api/platform/events fires events and notifies subscribers.

4.5 LLMRouter

File: src/platform/llm_router.py. Unified async chat / reason interface using LLMConfig (provider, chat model, optional reasoning model). Used by stack adapters and the AI wizard.

5. Agent stacks (adapters)

Defined in stacks/base.py; each adapter implements AgentStackAdapter: create_agent, invoke, start_loop, stop, get_status, scaffold_files.

Stack IDModuleTypical useScaffold shape
forgeos stacks/forgeos/adapter.py Default integration agent; uses platform LLM router and optional tool executor. agent.py, tools.py, prompts/system.md, config.yaml
crewai stacks/crewai/adapter.py Role/task/crew patterns; real CrewAI when SDK is importable, else LLM-router or simulated invoke. agents.py, tasks.py, crew.py, tools.py, config.yaml
adk stacks/adk/adapter.py Enterprise-style hierarchical workflows; scaffold comments for future Google ADK SDK. agent.py, workflow.py, tools.py, prompts, config.yaml
openclaw stacks/openclaw/adapter.py File-first: SOUL, identity, heartbeat, skills, memory layout; good for long-running personal automations. SOUL.md, IDENTITY.md, HEARTBEAT.md, SKILLS/, MEMORY/, gateway.sh

6. Execution types (lifecycle)

Enumeration ExecutionType in stacks/base.py. Wiring in PlatformExecutor._wire_execution:

ValueBehaviour
always_on Adapter start_loop: repeated invoke at configurable interval (metadata.loop_interval_seconds or stack defaults like OpenClaw heartbeat).
scheduled SchedulerEngine.add_job(agent_id, schedule, callback); callback invokes the agent with a schedule-oriented prompt.
event_driven For each trigger in event_triggers, subscribe on platform EventBus; on fire, invoke with event context.
reflex No automatic loop; agent is idle until explicitly invoked via API or UI “Test invoke”.
autonomous Asyncio task that repeatedly invokes until status COMPLETED/FAILED, iterations capped by metadata.max_iterations, delay metadata.loop_interval_seconds.

7. Ownership and on-disk layout

OwnershipType.PERSONAL: files under agents/personal/{owner_id}/{agent_name}/. OwnershipType.SHARED (enterprise / team): agents/shared/{agent_name}/.

Default AGENTS_ROOT is Path("agents") relative to the process working directory. The repository .gitignore typically excludes agents/personal/ and agents/shared/ so generated scaffolds are not committed.

8. Deploy / invoke / stop / undeploy

  1. Deploy — Build directory, write scaffolds, register, adapter create, wire execution.
  2. Invoke — Single turn through adapter (may call LLM router or external SDK).
  3. Stop — Cancel loops and jobs; status stopped; files remain.
  4. Undeploy (DELETE) — Stop, delete tree, unregister.

Legacy routes (/api/approvals, /api/workflows, etc.) serve the original company dashboard data model.

10. Next.js dashboard

Location: dashboard/. App Router pages include overview, agent list and filters, manual multi-step create, AI wizard, agent detail (including Test invoke), workflows, approvals, settings.

next.config.js rewrites /api/:path* to http://localhost:5000/api/:path* so browser calls to the Next origin reach Flask during development.

11. AI-assisted creation wizard

Backend: src/platform/agent_wizard_planner.py. Uses a fixed system rubric so the model outputs JSON with assistant_message, optional proposal, clarifying_questions, and ready_to_deploy. Proposals pass through normalize_proposal (valid enums, enterprise → shared, defaults for schedule/triggers/goals).

If no LLM API client is configured on LLMRouter, a heuristic mode infers stack and execution type from keywords so the UI remains usable offline.

12. Legacy company subsystem

Pre-platform ForgeOS behaviour: large agent registry per company module (src/companies/*/agent_configs.py), AgentInvoker, hooks, HITL gateway, metrics, MCP tools. These coexist with the platform: legacy agents are not the same list as platform_registry, but both run under one bootstrap and one Flask app.

14. Environment and operations

  • ANTHROPIC_API_KEY / OPENAI_API_KEY — enable real LLM paths in LLMRouter and richer wizard.
  • DATABASE_URL — PostgreSQL for production-oriented features; otherwise in-memory paths.
  • REDIS_URL — optional, hooks / rate limiting integration.
  • FORGEOS_SEED_HITL — set to 0/false/no to skip demo approval seeding.
  • Run tests: PYTHONPATH=. python -m pytest from repo root with venv activated.
