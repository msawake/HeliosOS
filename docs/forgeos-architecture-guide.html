ForgeOS Architecture Guide

How the Runtime, Kernel, and Syscall Pipeline enforce governance across all 5 adapters

1. System Overview

ForgeOS boots in a single sequence. Every agent invocation flows through the same governance stack regardless of which framework runs it.

Boot & Wiring Sequence
bootstrap.py → Kernel(registry, audit) → PlatformExecutor(registry, process_table, checkpoint_store) → runtime.register_platform(kernel, process_table, checkpoint_store)
ForgeOS Adapter CrewAI Adapter ADK Adapter OpenClaw Adapter Sandbox Adapter ← executor.register_adapter()
Every Agent Invocation
API / Dashboard / CLI → executor.invoke(agent_id, prompt) → runtime.bind(agent_id, namespace) → adapter.invoke() → Kernel gates every tool call → AgentResult

The executor binds the runtime before delegating to the adapter and unbinds after — so every tool call inside any adapter has access to the kernel.

2. Runtime → Kernel → Syscall Pipeline

The SDK runtime is the agent's interface. It talks to the Kernel facade, which composes 6 subsystems. For unified admission, the SyscallPipeline runs 7 ordered stages.

Three Layers
Agent Code
from forgeos_sdk import runtime → SDK Runtime
runtime.check_tool() → Kernel Facade
kernel.check_tool_call()
6 subsystems → Syscall Pipeline
kernel.syscall()
7 ordered stages

Kernel Subsystems

AdmissionController
Validates agent contracts before deploy. Name format, namespace uniqueness, tool availability, lifecycle consistency, dependency resolution. Also registers typed A2A surfaces.
PermissionManager
Runtime allow/deny for tool calls (wildcard whitelist + explicit deny list), A2A calls (ACL or same-namespace default), and data access.
BudgetManager
Per-task + daily USD caps. Two-phase reservation: reserve() → commit(actual) or release(). Prevents concurrent burst bypass.
PolicyEngine
Declarative JSON-logic rules from agent manifests. Operators: equals, contains, gt, in. Evaluates against a context dict.
DataBoundaryManager
Namespace isolation + PII policy. Checks allowed_namespaces / blocked_namespaces. Returns PII policy (allow/detect/mask/redact/block).
CapabilityManager
Opaque 128-bit tokens: issue(subject, target, verb, ttl). Positive authority that bypasses ACLs. Revocable, expiring.

Syscall Pipeline Stages

Fixed order. Each stage returns None (continue) or a KernelDecision (short-circuit on non-allow).

3. OpenClaw Developer View

OpenClaw agents run inside a Node.js gateway subprocess. Tool calls are proxied through a local HTTP server (ToolProxyServer) that validates tokens and checks kernel permissions before execution.

Tool Call Flow (Gateway Path)
SOUL.md
(LLM decides to call tool) → OpenClaw Gateway
(Node.js) → POST /tool
X-Agent-Token: sbx_... → ToolProxyServer
:18790 → Token verify →
runtime.check_tool() → tool_executor
.execute() → JSON result
What Happens on Denial
Agent calls shell.exec → POST /tool → check_tool("shell.exec") → {"error": "Kernel denied: not in allowed tools"}

The tool_executor is never reached. The proxy returns the error to the OpenClaw gateway, which passes it back to the LLM as a tool result.

4. Google ADK Developer View

ADK agents run via Runner.run_async(). Each ForgeOS tool is wrapped as an ADK FunctionTool with a kernel gate inside the async wrapper. The gate checks permissions before executing.

Tool Call Flow (Real ADK Path)
LlmAgent
(Gemini/Claude) → Runner.run_async()
event stream → FunctionTool
async _wrapper() → runtime.check_tool()
(kernel gate) → PermissionManager
+ BudgetManager
→ → → → → If allowed ↓ → tool_executor.execute() → result dict → ADK

The Kernel Gate (inside your FunctionTool wrapper)

# stacks/adk/adapter.py — _build_adk_tools()

async def _wrapper(**kwargs):
    # 1. Kernel gate: check permissions
    from src.forgeos_sdk.runtime import runtime as _rt
    if _rt.is_registered and _rt.is_bound:
        decision = await _rt.check_tool(tool_name, kwargs)
        if decision.denied:
            return {"success": False, "error": f"Kernel denied: {decision.reason}"}

    # 2. Execute tool (only if kernel allows)
    result = await tool_executor.execute(tool_name, kwargs, agent_context)
    return result

5. CrewAI Developer View

CrewAI agents run via Crew.kickoff() in a worker thread. Each ForgeOS tool is wrapped as a CrewAI BaseTool subclass with a kernel gate inside _run(). Because CrewAI runs synchronously in a thread, the gate creates its own event loop for the async kernel check.

Tool Call Flow (Real CrewAI Path)
Crew.kickoff()
(worker thread) → CrewAgent
(role, goal, backstory) → BaseTool._run()
(ForgeOSTool) → runtime.check_tool()
(new event loop) → Kernel
→ → → → If allowed ↓ → tool_executor.execute() → str result → CrewAI

The Kernel Gate (inside BaseTool._run)

# stacks/crewai/adapter.py — _build_crewai_tools()

class ForgeOSTool(CrewBaseTool):
    name: str = "email.send"
    description: str = "Send email"

    def _run(self, **kwargs) -> str:
        loop = asyncio.new_event_loop()
        try:
            # 1. Kernel gate
            from src.forgeos_sdk.runtime import runtime as _rt
            if _rt.is_registered and _rt.is_bound:
                decision = loop.run_until_complete(
                    _rt.check_tool(self.name, kwargs)
                )
                if decision.denied:
                    return f"Error: Kernel denied: {decision.reason}"

            # 2. Execute tool
            result = loop.run_until_complete(
                tool_executor.execute(self.name, kwargs, agent_context)
            )
        finally:
            loop.close()
        return str(result)

What Gets Checked on Every Tool Call
1. Is the tool in the allowed list?
Wildcard matching: mcp__fs__* matches mcp__fs__read
2. Is the tool in the deny list?
Explicit deny overrides allow
3. Is there budget remaining?
Per-task USD limit checked if cost estimate provided
4. Do policies permit it?
JSON-logic rules evaluated against tool name + input
5. Is the audit trail updated?
Every allow + deny recorded with context
