The 7 Components Every Agent Passes Through

Every agent invocation — whether a Simple Reflex checking emails or an Autonomous agent running a learning loop — flows through the same 7-step pipeline. Click each step to expand.

1
AgentInvoker
src/core/agent_invoker.py

The universal entry point. invoke(agent_id, prompt, task_metadata) does:

1. Registry lookup — finds the AgentConfig (model, tier, tools, system prompt)
2. Session create — generates UUID, registers as RUNNING
3. Audit log — records "agent_start" event
4. Dispatch — calls await _execute_agent() which enters the agentic loop
5. Error handling — catches TimeoutError, generic exceptions, updates status

Also provides delegate_to_subagent() (single task with retry) and delegate_parallel() (multiple tasks via asyncio.gather()).

2
HookChain
src/core/hooks.py

7 governance checks run on every tool call, in order:

1. Budget Pre-Check — estimates cost of next API call, BLOCKS if over limit
2. Rate Limiter — per-session count + per-minute sliding window
3. Auth Check — validates tool is in agent's allowed_tools list
4. Cost Tracker — tracks running session cost
5. Compliance Checker — scans outbound emails for PII, invalid claims
6. Slack Notifier — alerts humans on critical events
7. Audit Logger — immutable record of every action

Each hook returns: ALLOW, BLOCK, or ASK_HUMAN. If any hook blocks, the tool call is denied.

3
Async/Sync Bridge
src/core/claude_client.py

asyncio.to_thread() moves the agentic loop into a worker thread. This is critical because:

• The LLM API client is synchronous — each call blocks for 2-30 seconds
• If it ran on the event loop, ALL other agents would freeze
• Worker threads let the event loop continue running other agents

When the agentic loop (in the thread) needs to call an async tool, it bridges back via asyncio.run_coroutine_threadsafe().

4
Agentic Loop (Reason → Act → Observe)
src/core/claude_client.py → _run_sync()

The core loop repeats up to 50 turns:

1. Reason — Call LLM with conversation history. LLM produces text + tool calls
2. Act — For each tool call: run HookChain pre-checks → execute tool → run post-hooks
3. Observe — Append tool results to conversation, checkpoint session
4. Repeat — Until LLM returns stop_reason: "end_turn" (no more tool calls)

LLM API calls use exponential backoff retry: 1s → 2s → 4s, max 3 attempts, capped at 30s.

5
ToolExecutor
src/mcp/tool_executor.py

Routes tool calls by name prefix:

• company__* → in-process handlers: EventBus, HITLGateway, KnowledgeBase, MetricsStore
• mcp__* → external MCP servers: parsed as mcp__<server>__<method>, dispatched to the connected MCP client
• Unknown → returns error

Example: mcp__google-workspace__send_gmail_message → Google Workspace MCP server → send_gmail_message method.

6
SessionStore
src/core/session_store.py

Saves full conversation state after every turn:

• messages — complete conversation history
• turns_completed, tool_calls_completed
• input_tokens, output_tokens, cost_usd
• last_checkpoint_at — ISO timestamp

Two backends: InMemorySessionStore (default, dict-based) and PostgresSessionStore (production, survives restarts). If the process crashes mid-conversation, the agent can resume from the last checkpoint.

7
AgentResult
src/core/agent_invoker.py

The final output returned to the caller:

• status — completed / failed / timeout
• result — the agent's final text response
• tokens_consumed — total input + output tokens
• cost_usd — estimated dollar cost
• duration — wall-clock seconds
• tool_calls — count of tools executed
• error — error message if failed

The Async/Sync Bridge — Why It Exists

Problem

The event loop runs all agents concurrently. But the LLM API is synchronous — each call blocks for 2-30 seconds. If called directly on the event loop, every other agent freezes.

Solution

asyncio.to_thread() runs the agentic loop in a worker thread. The event loop yields immediately and continues running other agents while the LLM call blocks in its own thread.

But... Tools Are Async

The ToolExecutor is async (MCP calls, DB queries). The worker thread bridges back to the event loop via asyncio.run_coroutine_threadsafe(), waits for the result, then continues.

The Three Code Layers

# Step 1: Invoker calls ClaudeClient (async → spawns thread) async def run(self, ...): return await asyncio.to_thread(self._run_sync, ...) # → worker thread # Step 2: Agentic loop runs in worker thread (sync, blocking OK) def _run_sync(self, ...): for turn in range(max_turns): response = self._call_llm_with_retry(...) # blocking — fine in thread for tool_call in response.tool_calls: result = self._execute_tool(...) # needs async bridge ↓ # Step 3: Tool execution bridges back to event loop def _execute_tool(self, tool_name, tool_input, ctx): return _run_async_from_thread( # sync thread → async loop self._executor.execute(tool_name, tool_input, ctx) )

auto-responder Simple Reflex Always-On Haiku LeadForge

Receives inbound email inquiries, sends canned acknowledgment. IF inbound email → THEN send template. No scoring, no personalization.

Execution Timeline

0
Bootstrap: asyncio.create_task()
During boot Phase 6, _start_standing_swarms() creates an asyncio task: asyncio.create_task(_run_standing_agent("auto-responder", "Auto-Responder", 0)). This runs in a while self._running loop, invoking the agent every 60 seconds.
1
Turn 1 — Reason: Check for pending emails
Haiku receives "Execute your standing duties..." prompt. System prompt says: "You respond to inbound emails with acknowledgment templates." Haiku calls company__get_pending_events(type="email"). HookChain allows (cheap agent, low risk). ToolExecutor routes to EventBus → returns 1 pending email.
2
Turn 2 — Act: Send acknowledgment
Haiku sees 1 pending email, decides to send template response. Calls company__publish_event(type="email_response", payload={to: "hello@...", body: "Thanks, an SDR will follow up..."}). HookChain allows. EventBus publishes successfully.
3
Turn 3 — Done
Haiku says "Done. Processed 1 inbound inquiry." stop_reason: "end_turn" — exits the agentic loop. SessionStore saves final state. Returns AgentResult(status=COMPLETED, cost=$0.002).
∞
Sleep 60s → Next cycle
await asyncio.sleep(60) — yields control to the event loop. Other agents run during this time. After 60 seconds, the cycle repeats.
Key insight: Simple Reflex agents finish in 2-3 turns (~2 seconds). Haiku is fast and cheap ($0.002/cycle). The 60-second sleep between cycles is where asyncio runs all the other agents concurrently.
lead-scorer Model-Based Reflex Event-Triggered Sonnet LeadForge

Scores leads using the BANT framework. Accumulates signals across interactions into a persistent lead profile. Flags leads as SQL when score ≥70 with ≥2 qualification signals.

Execution Timeline

E
Event: Prospect opened pricing email
EventBus receives engagement signal → triggers workflow task. WorkflowEngine dispatches: invoker.invoke(agent_id="lead-scorer", prompt="Score lead #L-4521. New signal: email_open on pricing email.") with TaskMetadata(budget_tokens=5000).
1
Turn 1 — Read existing lead profile
Sonnet calls company__search_knowledge("lead profile L-4521"). KnowledgeBase returns existing state: {score: 55, signals: {budget: 15, authority: 10, need: 20, timeline: 10}}. This is the "model" in Model-Based — accumulated state from prior invocations.
2
Turn 2 — Apply BANT scoring, update state
Sonnet reasons: "Email open on pricing page = budget signal (+5) and timeline signal (+5). New score: 55 + 10 = 65. Qualification signals: budget ✓, need ✓ = 2. Below 70 threshold — flag as MQL." Calls company__add_decision(category="lead_scoring", key="L-4521", value={score: 65, status: "MQL"}).
3
Turn 3 — Record metric
company__record_metric(name="leads.scored", value=1, labels={score: 65, status: "MQL"})
4
Turn 4 — Done
"Lead L-4521 scored at 65 (MQL). One more engagement should trigger SQL threshold." Cost: $0.015, 4 turns.
Key insight: State persists in KnowledgeBase, not in agent memory. The agent read score 55 (from a previous invocation), applied +10, wrote 65. Next time this lead triggers an event, the agent picks up from 65. The "model" survives across invocations because it's externalized.
campaign-builder Goal-Based On-Demand / HITL Sonnet LeadForge

Goal: "Launch outbound campaign for [ICP]." Plans the full sequence: research → prospect list → email sequences → human approval → schedule sends. Can delegate to sub-agents (Tier 2 = DEPARTMENT_LEAD).

Execution Timeline

H
Human triggers: "Launch campaign for fintech CFOs"
Operator uses dashboard or Slack. WorkflowEngine dispatches with TaskMetadata(budget_tokens=50000, priority="high"). Registry lookup: Sonnet, DEPARTMENT_LEAD, sales. SUBAGENT_MAP["campaign-builder"] = ["sales-researcher", "sales-sdr"] — this agent CAN delegate.
1
Turn 1 — Plan: Break goal into steps
Sonnet plans: 1) Research ICP → delegate to sales-researcher, 2) Build prospect list, 3) Write email sequences, 4) Get human approval via HITL, 5) Schedule sends → delegate to sales-sdr. Calls company__search_knowledge("ICP fintech CFO") for templates.
2
Turn 2 — Delegate: Spawn sales-researcher sub-agent
Calls company__invoke_subagent(agent_id="sales-researcher", prompt="Research fintech companies Series B+. Find 50 CFO contacts."). This triggers a nested AgentInvoker.invoke() — the sub-agent runs its own agentic loop (4 turns: search → filter → compile → return). sales-researcher is Tier 3 WORKER — it CANNOT delegate further. Returns: "Found 47 CFOs matching criteria."
3
Turn 3 — Write 3-email sequence
Sonnet writes: Email 1 (cold intro, personalized), Email 2 (value prop follow-up, Day 3), Email 3 (social proof + CTA, Day 7).
4
Turn 4 — HITL Gate: Request human approval
Calls company__request_approval(category="content", description="Email sequence for 47 fintech CFOs", sla_hours=4.0). HITLGateway creates approval request. Slack notification sent. Dashboard shows pending item. AGENT PAUSES HERE. Returns: "Awaiting approval APR-7823."
✓
Human approves → New invocation
Operator reviews emails in dashboard, clicks approve. EventBus fires approval_granted. WorkflowEngine dispatches NEW invocation of campaign-builder with prompt "Approval granted. Proceed with scheduling." Knowledge persists in KnowledgeBase across invocations.
5
Turn 5 — Delegate: Send via sales-sdr
Delegates to sales-sdr: "Schedule 3-email cadence for 47 contacts. Max 50 emails/SDR/day (CAN-SPAM)." Sub-agent executes, returns success. Final: "Campaign launched. First batch sends tomorrow at 9 AM EST."
Key insight: Three critical patterns: Delegation (Tier 2 spawns Tier 3 sub-agents as nested invoke() calls), HITL pause/resume (agent stops at approval gate, new invocation continues after human approves), Tier enforcement (Tier 3 workers cannot delegate further).
budget-optimizer Utility-Based Scheduled Opus LeadForge

Allocates daily marketing spend across channels (Google Ads, LinkedIn, email) to maximize cost-per-qualified-lead. Weighs channel performance, remaining budget, and constraints.

Execution Timeline

⏰
Scheduled trigger: 6:00 AM daily
Main loop tick detects scheduled task is ready. HookChain pre-check: CostTracker estimates $0.45 for Opus/30K tokens. Session cost $0.00, limit $50.00 → ALLOW.
1
Turn 1 — Gather yesterday's channel performance
company__get_metric("marketing.channel_performance") → Google Ads: $450 spend, 4 qualified, CPL $112.50 | LinkedIn: $300, 5 qualified, CPL $60.00 | Email: $50, 3 qualified, CPL $16.67
2
Turn 2 — Gather budget constraints
company__get_metric("marketing.daily_budget") → total: $800, remaining this month: $12,400
3
Turn 3 — Reason: Utility function optimization
Opus analyzes tradeoffs:

LinkedIn: Best CPL ($60) but saturates at ~$400/day
Email: Cheapest ($16.67) but at capacity (50 emails/SDR × 3 SDRs)
Google Ads: Expensive ($112.50) and Quality Score dropped

Optimization: Maximize qualified_leads = f(spend_per_channel), constrained to total ≤ $800.
Result: LinkedIn $400 (+$100), Google Ads $300 (-$150), Email $50 (same), Reserve $50.
Expected: 14 qualified leads (vs 12 yesterday), CPL $57.14 (vs $66.67).
4
Turns 4-6 — Record + Publish
Turn 4: company__add_decision (save allocation to KnowledgeBase)
Turn 5: company__record_metric (track allocation in MetricsStore)
Turn 6: company__publish_event (notify marketing dept of reallocation)
7
Turn 7 — Done. Cost: $0.38
"Daily budget optimized. Shifted $100 from Google Ads to LinkedIn based on CPL analysis. Expected: +2 qualified leads, -14% CPL." No HITL gate needed — reallocation is under $1K threshold.
Key insight: Utility-Based agents don't just execute a plan — they evaluate multiple options against a utility function (cost-per-qualified-lead) and pick the optimal one. Opus is needed for the multi-factor tradeoff reasoning. Scheduled daily because budget decisions are daily cycles.
sdr-autonomous Autonomous Always-On Opus LeadForge

Full sales development loop: research → personalize outreach → send → track responses → adjust messaging → re-target. Self-tunes email templates based on reply rates. 3 concurrent instances.

Execution Timeline (Instance 0 — West Coast territory)

⚡
Bootstrap: 3 instances via asyncio.create_task()
Three concurrent instances launched: Instance 0 (West Coast), Instance 1 (Midwest), Instance 2 (East Coast). Each runs in its own asyncio task — they don't block each other. While one waits for an LLM call, the event loop runs the others.
1
OBSERVE — Turn 1: Load current state
company__search_knowledge("sdr-autonomous-0 state") → Returns: territory: west_coast, 23 active sequences, template performance: "direct-ask" 12% reply, "value-first" 17.6%, "social-proof" 8.3%.
2
OBSERVE — Turn 2: Check engagement signals
company__get_pending_events(type="engagement") → 3 signals: L-8821 replied positively, L-7703 visited pricing page, L-9102 email bounced.
3
REFLECT — Turn 3: Analyze performance + decide adjustments
Opus reflects: "value-first has 17.6% vs social-proof at 8.3% → retire social-proof. L-8821 replied positively → hand off to AE. L-7703 pricing visit → score them. L-9102 bounced → remove." ADJUSTMENT: Shift all new prospects to value-first template. Create value-first-v2 variant with shorter subject line.
4
ACT — Turns 4-8: Execute adjustments
Turn 4: Update strategy in KnowledgeBase (retire social-proof, activate value-first)
Turn 5: Hand off L-8821 to sales-ae via EventBus
Turn 6: Trigger lead scoring for L-7703
Turn 7: Mark L-9102 as invalid (bounce)
Turn 8: Request HITL approval for 5 new outreach emails (external action requires human approval in supervised mode)
9
Turn 9-10 — Record metrics + Done. Cost: $0.52
"Cycle complete. Retired social-proof template (8.3%). Doubled down on value-first (17.6%). Handed off L-8821. 5 new outreach pending approval." Then await asyncio.sleep(60) — next cycle picks up from the UPDATED strategy.
∞
Cycle 2, 3, 4... Strategy evolves over weeks
Each cycle reads the latest state, observes results, reflects, adjusts. Over days and weeks: templates get refined, reply rates improve, underperforming approaches get retired automatically. The agent gets better at its job without human intervention — that's what makes it Autonomous.
Key insight: Autonomous = learning loop. Each cycle: observe (signals) → reflect (analyze what's working) → adjust (change strategy) → act (execute). State persists in KnowledgeBase across cycles. Even in autonomous mode, external actions (emails) still go through HITL governance. Opus is required for the reflection reasoning.

How Concurrency Actually Works

At any given moment, the event loop manages all agents simultaneously. Nothing on the event loop ever blocks — LLM calls run in worker threads, asyncio.sleep() yields control.

EVENT LOOP STATE Tick: 0
● sdr-autonomous-0
RUNNING Turn 5
● auto-responder-0
SLEEP 42s
● sales-sdr-0
SLEEP 18s
● sales-sdr-1
SLEEP 55s
● ops-monitoring
SLEEP 33s
● sdr-autonomous-1
SLEEP 7s

The animation above shows agents cycling through states. RUNNING = in a worker thread making LLM calls. SLEEPING = awaiting next cycle (asyncio.sleep(), yielding to the event loop). When a sleeping agent's timer expires, it wakes up and starts a new cycle — all without blocking any other agent.
